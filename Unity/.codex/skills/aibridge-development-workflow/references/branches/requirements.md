# 需求讨论分支

## 适用场景

当用户目标、边界、非目标、约束、验收标准或方案方向还没有锁定时，先进入需求讨论分支。

典型触发信号：

- 用户明确要求先分析、先讨论、先确认、先出方案。
- 同一目标存在多个合理实现方向。
- 需求边界、数据来源、兼容性约束或验收标准不明确。
- 任务需要先写入项目文档，再进入实施。

## 进入规则

1. 这是 Preflight 的前置模式，不是实施分支的替代品。
2. 先把问题收敛成可确认的方案，再重新执行 Preflight / Skill 路由。
3. 方案确认前，不进入实施分支。

## 执行规则

- 先确认目标、边界、非目标、约束和验收标准。
- 若有多个合理方案，先列出可选项和取舍，再锁定一个。
- 所有方案默认先落到 `.aibridge/plan` 作为工作底稿和上下文锚点；需求讨论阶段可先写 `draft`，方案确认后再更新为 `confirmed`。
- 如果用户要求，或者项目有相应的功能文档归类，再把同一份方案同步到对应正式文档位置。
- 默认先写 Markdown；当方案需要流程图、决策树、对比表，或更强的开发者浏览效果时，再在每个落点目录内同步生成 HTML 展示页。
- `.aibridge/plan` 负责 AI 续跑和多 agent 协作，正式文档负责 Git review 和对外呈现。

## 输出

建议输出一个可执行的方案摘要，至少包含：

- 目标
- 边界
- 非目标
- 约束
- 方案选项
- 已确认决策
- 待确认事项
- 工作底稿路径（`.aibridge/plan/...`）
- 正式文档路径（如适用）
- 展示文档路径（如适用）

## Handoff

确认完成后，输出 compact handoff，再转入正式主分支。
