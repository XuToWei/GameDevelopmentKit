# 需求讨论分支、方案确认与风险门控

## 无需确认

满足以下条件时可以直接实施，并在入口说明无需进入需求讨论分支：

1. 用户目标明确，验收结果可以从上下文推断。
2. 修改范围小，主要影响单个文件、局部逻辑或文档。
3. 不改变公共 API、序列化数据结构、配置格式、资源路径或用户可见行为。
4. 不删除资源、不迁移数据、不引入新依赖。

## 何时进入需求讨论分支

出现任一情况时，先进入需求讨论分支，不直接进入实施分支：

1. 用户需求存在多种合理理解。
2. 目标、边界、非目标、约束或验收标准没有锁定。
3. 需要新增依赖、改变包结构、迁移配置或修改公共协议。
4. 会影响多个系统、公共 API、Prefab/场景结构或已有资源引用。
5. 涉及删除、覆盖、批量写入、不可逆修改。
6. 用户明确要求“先分析”“先给方案”“先讨论”“不要直接改”。

## 复杂度评估

- **常规问题**：已有模式清晰、影响范围有限，直接实施。
- **复杂问题**：涉及架构决策、跨模块影响、兼容性风险，先确认方案。
- **非常规问题**：现有模式不适用或需要创造性设计，先给多个方案和取舍。

## 方案写入规则

方案确认后，若用户有要求，或者项目中有相应的功能文档归类，必须先更新 `.aibridge/plan` 中的工作底稿，再同步到正式文档位置。未确认前只允许停留在需求讨论分支输出或 `.aibridge/plan` 的 draft，不进入正式方案发布。

1. 默认先写 `.aibridge/plan/<slug>.md`，它是方案的工作底稿和上下文锚点。
2. 当方案包含流程图、决策树、对比表，或需要更强的开发者展示效果时，再在每个落点目录内同步写 `html`。
3. 若存在正式文档位置，再同步写入对应目录，保持与工作底稿同 basename。
4. `html` 是展示副本，不替代 `md` 作为可审阅版本。
5. 如果只写一种格式，默认写 `md` 工作底稿。

## 需求讨论分支输出

```text
【需求讨论分支】
方案概述：...
需求边界：...
非目标：...
约束：...
已确认决策：...
待确认事项：...
工作底稿：.aibridge/plan/xxx.md
正式文档：.../xxx.md（如适用）
展示文档：.../xxx.html（如适用）

请确认后继续。
```

## 输出建议

```text
待确认事项：
1. ...
2. ...

请确认后继续。
```
