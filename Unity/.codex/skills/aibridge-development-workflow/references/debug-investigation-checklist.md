# 调试诊断检查清单

## 选择规则

调试诊断分支结束前执行本清单。按任务实际情况标记 `✅`、`❌`、`已补充` 或 `不适用`。

## 问题定界

1. 已记录症状、期望结果、实际结果和触发条件。
2. 已记录环境：Unity Editor/Play Mode/Player、平台、Runtime target id 或 URL。
3. 已明确用户当前目标是诊断、验证还是允许进入修复。

## 证据完整性

1. Editor 编译或 Console 证据已收集；无法执行时已说明原因。
2. Runtime/Player 相关任务已收集 target 状态、日志和必要截图/性能/handler 证据。
3. 每条关键证据都保留了命令、输出摘要、artifact 路径或代码位置。
4. 证据时间线与复现步骤一致；没有使用明显过期的日志或截图支撑结论。

## 分析严谨性

1. 每个候选根因都标记为 `confirmed`、`refuted` 或 `uncertain`。
2. confirmed 根因有可重复证据，不只依赖猜测或单条弱信号。
3. uncertain 项已说明缺少什么证据以及下一步如何补证。
4. Runtime 错误被当作诊断证据处理；未把“存在错误”误判为 workflow 本身失败。

## 修改门控

1. 用户只要求排查时，没有直接修改代码、资源、Prefab、Scene 或配置。
2. 如果已进入修复，已把 confirmed 根因、证据和修改范围交接到实施分支。
3. 修复后已执行实施分支检查清单和必要 Runtime/UI 回归验证。

## 输出格式

```text
【模式：分支检查清单】
主分支：调试诊断分支
1. 问题定界 ✅
2. Editor/Console 证据 ✅
3. Runtime target 证据 ✅
4. 候选根因 verdict ✅
5. 修改门控 ✅（本次只诊断，未修改代码）
6. 未验证项：Android 真机复现（当前无设备）

调试诊断完成：根因状态 confirmed，证据和剩余风险已列出。
```
