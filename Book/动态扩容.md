# 动态扩容方案

## 一、现状分析

### 当前配置链路

```
Excel 手动配置 IP/Port
        ↓
StartMachineConfig  →  {Id, InnerIP, OuterIP, WatcherPort}
StartProcessConfig  →  {Id, MachineId, Port}         → 通过 MachineId 引用 Machine 的 IP
StartSceneConfig    →  {Id, Process, Zone, SceneType, Port} → 通过 Process 引用 Process 的 IP
        ↓
Aspire 读取配置，逐个启动 Process（Program.cs）
        ↓
每个 Process 读取自己的 Scene 列表，创建 Fiber（EntryEvent2_InitServer.cs）
        ↓
跨进程通信时，通过 ProcessId 查 DTStartProcessConfig 拿 IP:Port（ProcessOuterSenderSystem.cs:139）
```

### 核心问题

1. **手动配 IP/Port**：每加一个 Gate/Map 都要手动填 Excel，容易出错
2. **静态地址查表**：`ProcessOuterSenderSystem.Get()` 直接查 `DTStartProcessConfig` 拿地址，运行时不可变
3. **Gate 取模分配**：`hash % gateCount`，扩容导致全量 rehash
4. **Location 单点**：`LocationConfig` 指向唯一 Location Fiber
5. **无法运行时增删 Scene**

## 二、配置简化

### 2.1 目标

**之前**：手动配置每个 Machine、Process、Scene 的 IP/Port

**之后**：保留三张配置表的结构，删掉 Port 字段（由系统自动分配），Scene 表加 Count 字段

### 2.2 配置表改造

保留 StartMachineConfig、StartProcessConfig、StartSceneConfig，只做字段增删：

**StartMachineConfig**（保留）：

| 字段 | 变化 |
|------|------|
| Id | 保留 |
| InnerIP | 保留（多机部署需要） |
| OuterIP | 保留（多机部署需要） |
| WatcherPort | 删除（Watcher 机制被 Aspire 替代） |

**StartProcessConfig**（保留）：

| 字段 | 变化 |
|------|------|
| Id | 保留 |
| MachineId | 保留（指定 Process 跑在哪台机器） |
| Port | 删除（由系统自动分配） |

**StartSceneConfig**（保留 + 新增 Count）：

| 字段 | 变化 |
|------|------|
| Id | 保留 |
| Process | 保留（指定 Scene 归属哪个 Process，0 表示自动分配） |
| Zone | 保留 |
| SceneType | 保留 |
| Name | 保留 |
| Port | 删除（由系统自动分配） |
| **Count** | **新增**（该类型 Scene 启动的数量，默认 1） |

**示例**：把 Gate 的 Count 从 2 改成 5，Aspire 就自动启动 5 个 Gate Process。

StartZoneConfig（Zone → DB 连接）保持不变。

### 2.3 Aspire 自动生成拓扑

Aspire `Program.cs` 改造：读取三张配置表，根据 Count 字段自动展开，不分配端口。

```
输入: StartSceneConfig（含 Count 字段）+ StartProcessConfig + StartMachineConfig
      ↓
展开:
  Count > 1 的 Scene，自动生成多个实例（SceneId 递增）
  Process 字段为 0 的 Scene，自动分配到可用 Process
      ↓
输出: 启动 Registry + 对应数量的 Process
```

**端口分配**：系统级自动获取，不预分配。

每个 Process 启动时绑定端口 0，由 OS 分配可用端口，然后将实际端口注册到 Registry：

```
Process 启动 → KService 绑定 port 0 → OS 分配实际端口（如 52341）
            → 读回实际端口 → 向 Registry 注册 (ProcessId=5, IPEndPoint=127.0.0.1:52341)
```

Aspire 不需要管理端口，端口冲突问题彻底消失。

**唯一例外**：Registry 本身需要固定端口（如 20000），因为它是所有 Process 的发现入口。

**IP 来源**：从环境变量或命令行参数获取（单机默认 127.0.0.1，多机部署时由运维指定）。

### 2.4 进程启动参数变更

当前每个 Process 启动后自己查 `DTStartProcessConfig` 和 `DTStartSceneConfig` 获取自身信息。

改为 Aspire 通过启动参数 / 环境变量直接传递：

```
--Process=5
--SceneId=1004
--SceneType=Gate
--Zone=1
--RegistryAddress=127.0.0.1:20000
```

不再传 Port 参数，Process 自行绑定 port 0 获取系统分配的端口。

`EntryEvent2_InitServer` 从 Options 读取这些参数创建 Fiber，不再查 Excel 配置。

## 三、服务注册中心（ServiceRegistry）

### 3.1 为什么需要

配置简化后，每个 Process 知道自己的地址，但**不知道其他 Process 的地址**。

当前 `ProcessOuterSenderSystem` 第139行：

```csharp
IPEndPoint ipEndPoint = Tables.Instance.DTStartProcessConfig
    .Get(Options.Instance.StartConfig, (int)channelId).IPEndPoint;
```

这行代码假设所有 Process 地址都在静态配置里。删掉静态配置后，需要一个运行时的地址查询机制。

### 3.2 设计

新增 `SceneType.Registry`，作为第一个启动的 Process，固定地址（唯一需要预知地址的服务）。

**Registry 功能**：

- **注册**：每个 Process 启动后向 Registry 注册 `(ProcessId, IP, Port, SceneType, Zone)`
- **地址查询**：Process 需要连接其他 Process 时，向 Registry 查询地址
- **路由管理**：统一管理 Gate、Map、Location 等的路由策略（一致性哈希环、分片表），各 Process 不再各自硬编码路由逻辑
- **推送**：节点增减或路由变更时主动通知已订阅的 Process

**各 Process 启动流程**：

```
1. Process 启动
2. 创建 NetInner Fiber，绑定 port 0，OS 分配实际端口
3. NetInner 向 Registry 注册 (ProcessId, 实际 IPEndPoint, SceneType, Zone)
4. NetInner 拉取当前全量地址表，缓存到本地
5. 订阅地址变更通知
6. 创建业务 Scene Fiber
7. 就绪，开始处理消息
```

**Registry 地址如何获知**：

- Aspire 将 Registry 的地址作为环境变量传给所有 Process：`--RegistryAddress=127.0.0.1:20001`
- Registry 本身是固定端口，Aspire 最先启动它

### 3.3 改造 ProcessOuterSender 地址解析

```csharp
// 之前：静态查表
private static Session Get(this ProcessOuterSender self, long channelId)
{
    IPEndPoint ipEndPoint = Tables.Instance.DTStartProcessConfig
        .Get(Options.Instance.StartConfig, (int)channelId).IPEndPoint;
    ...
}

// 之后：从 Registry 本地缓存查
private static Session Get(this ProcessOuterSender self, long channelId)
{
    IPEndPoint ipEndPoint = self.GetAddressFromRegistry((int)channelId);
    ...
}
```

`GetAddressFromRegistry` 优先查本地缓存，未命中时同步查 Registry。

### 3.4 改造 DTStartSceneConfig 的辅助查询

当前 `DTStartSceneConfig` 在 PostInit 中建了多个索引（Gates、Maps、Realms 等），供各处查询。

改为由 Registry 提供等价查询：

| 当前查询 | 改造后 |
|----------|--------|
| `DTStartSceneConfig.Gates[zone]` | `Registry.GetScenes(zone, SceneType.Gate)` |
| `DTStartSceneConfig.GetBySceneName(zone, name)` | `Registry.GetScene(zone, name)` |
| `DTStartSceneConfig.LocationConfig` | `Registry.GetScenes(0, SceneType.Location)` |
| `DTStartSceneConfig.Maps` | `Registry.GetScenes(SceneType.Map)` |

## 四、ActorId 简化

### 当前结构

```
ActorId (16字节):
  Address:
    Process:    int (4字节)   — 进程ID
    Fiber:      int (4字节)   — 纤程ID
  InstanceId:   long (8字节)  — 实例ID
```

### 改造后

```
ActorId (8字节):
  FiberId:      int (4字节)   — 全局唯一纤程ID（集群内不重复）
  InstanceId:   int (4字节)   — 纤程内递增计数器
```

**大小从 16 字节降到 8 字节，每条跨进程消息省 8 字节。**

### 设计要点

**去掉 Process**：FiberId 全局唯一，Registry 维护 `FiberId → ProcessId → IPEndPoint` 映射。不再需要 Process 字段来定位目标。

**本地判断**：每个 Process 维护本地 Fiber 集合，替代当前的 Process 比较：

```csharp
// 之前
if (actorId.Process == fiber.Process) { /* 本地发送 */ }

// 之后
if (localFibers.Contains(actorId.FiberId)) { /* 本地发送 */ }
```

**InstanceId 改 int**：32 位递增计数器，每个 Fiber 独立，43 亿个值足够。InstanceId 只需在同一 Fiber 内不重复。

**FiberId 全局分配**：由 Registry 统一分配 FiberId，确保集群内不重复。动态创建的 Fiber 也向 Registry 申请 ID。

### 改造文件

**ET Core 层**：

| 文件 | 改动 |
|------|------|
| `ActorId.cs` | 去掉 Address 结构，改为 `int FiberId` + `int InstanceId` |
| `Entity.cs` | `long InstanceId` → `int InstanceId` |
| `EntityRef.cs` | `long instanceId` → `int instanceId`（EntityRef + EntityWeakRef） |
| `IdGenerater.cs` | `InstanceIdStruct` 去掉 Time 只留 32bit；`GenerateInstanceId()` 返回 int |
| `Scene.cs` | 构造函数参数 `long instanceId` → `int instanceId` |
| `Mailboxes.cs` | 参数 `long instanceId` → `int instanceId` |
| `MailBoxComponent.cs` | `long ParentInstanceId` → `int ParentInstanceId` |
| `Fiber.cs` | `GetActorId()` 不再需要 Process |
| `ByteHelper.cs` | ActorId 序列化从 16 字节改为 8 字节 |
| `MessageSerializeHelper.cs` | 反序列化适配新的 8 字节格式 |
| `MessageSenderSystem.cs` | 本地判断从 `actorId.Process ==` 改为 `localFibers.Contains()` |
| `ProcessOuterSenderSystem.cs` | 通过 Registry 查 FiberId 对应的 Process 地址 |
| `EntitySceneFactory.cs` | 参数类型同步 |
| `ComponentViewEditor.cs` | `LongField` → `IntField` |

**Proto 层**：

| 文件 | 改动 |
|------|------|
| `InnerMessage.proto` | `int64 InstanceId` → `int32 InstanceId` |
| `OuterMessage.proto` | `int64 SceneInstanceId` → `int32 SceneInstanceId` |

**业务代码**（跟编译错误改 `long` → `int`）：约 10 个文件。

## 五、Gate 一致性哈希

### 当前

```csharp
// RealmGateAddressHelper.cs
zoneGates[(int)(hash % (ulong)zoneGates.Count)]
```

Gate 数量从 2 变成 3 时，约 2/3 的玩家 hash 结果改变。

### 改造

引入一致性哈希环（每个 Gate 节点映射 N 个虚拟节点到环上）：

- 扩容时只迁移约 `1/N` 的玩家
- Gate 列表从 Registry 获取，节点变更时重建哈希环
- 同样适用于 Map 分配

## 五、Map 动态伸缩

### 扩容

1. Map Fiber 定期上报负载（玩家数）到 Registry
2. Registry 检测到某类 Map 过载
3. Aspire 启动新 Process，指定 SceneType=Map
4. 新 Process 注册到 Registry
5. 后续玩家进 Map 时负载均衡到新实例

### 缩容

1. 某 Map 长时间低负载
2. 将残余玩家通过 `TransferHelper` 迁移到同类其他 Map
3. 销毁 Fiber，从 Registry 摘除
4. Aspire 停止对应 Process

### 改造点

- `C2G_EnterMapHandler`：Map 选择从 `GetBySceneName` 改为从 Registry 查询 + 按负载选择
- `EntryEvent2_InitServer`：支持运行时接收 CreateScene 指令
- Aspire：暴露 HTTP API 供 Registry 触发启停 Process

## 六、Location 分片

### 当前

```csharp
// LocationProxyComponentSystem.cs
private static ActorId GetLocationSceneId(long key)
{
    return Tables.Instance.DTStartSceneConfig.LocationConfig.ActorId;
}
```

所有 key 打到同一个 Location Fiber，单点瓶颈。

### 改造

- 部署 N 个 Location Fiber（通过 StartSceneConfig 的 Count 字段配置）
- 路由由 Registry 统一管理（维护 key → Location 分片的路由表），推送到各 Process 本地缓存
- Location 扩缩容时，Registry 更新路由表并推送，不需要改代码

## 八、实施顺序

### 阶段一：配置简化 + Registry

1. 配置表改造（删 Port、加 Count）
2. 实现 ServiceRegistry（注册、查询、推送）
3. 改造 Aspire 根据 Count 展开实例并启动 Registry
4. 改造 ProcessOuterSender 地址解析走 Registry
5. 改造 EntryEvent2_InitServer 从 Options 读参数

**验收**：配置表里无 Port 字段，改 Count 即可增减实例数量，服务正常启动和通信。

### 阶段二：弹性扩容

1. Gate 一致性哈希
2. Location 分片
3. Map 负载上报 + 动态伸缩
4. Aspire 暴露 API 支持运行时启停 Process

**验收**：运行时增加 Gate/Map 数量，新玩家自动路由到新节点。
